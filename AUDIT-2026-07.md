# AUDIT-2026-07 — gopher-spot (fio A1)

Auditoria de qualidade **read-only** do servidor Rust (dcgi + pipeline de áudio),
executada no fio A1 antes de qualquer hardening. Objetivo: **encontrar, classificar
e priorizar — não consertar**. Nenhuma linha de código foi alterada; o único
artefato deste fio é este relatório.

Metodologia: leitura integral de `src/*.rs`, `docker/audio-stream-entrypoint.sh` e
`deploy/*.yaml`; sweep de `unwrap/expect/panic/index` e de `eprintln!/dbg!`; execução
real de `cargo test` (82 passam, 0 falham) e `cargo clippy --all-targets -- -W
clippy::pedantic`; triagem manual de `Cargo.lock`. Todo achado P1 e todo achado de
segurança foram **verificados adversarialmente** por um segundo leitor (marcados
`[verificado]`).

## Fase 0 — Exposição (calibra severidade)

`:70` (gopher-server) e `:8000` (audio-stream) são `type: LoadBalancer` na pool
MetalLB `10.0.100.x`, **sem `loadBalancerSourceRanges` e sem `NetworkPolicy`**. Ou
seja: na borda k8s ambos estão abertos a qualquer coisa que alcance a pool. Pela
topologia do projeto essa pool é **LAN/WireGuard-only (RFC1918), não pública**, e o
serviço é single-user homelab acessando **a própria conta** Spotify do Felipe. Isso
**limita toda a superfície abaixo a um atacante já-na-LAN** e é o motivo de nenhum
achado chegar a P0/P1-de-segurança. A auditoria mesmo assim leu cada byte do cliente
como hostil.

**Veredito path traversal (item de DoD): REFUTADO — impossível por construção.**
`cache.rs:17-24` (`key_file`) dobra toda chave com FNV-1a/64 e nomeia o arquivo
`dir.join(format!("{h:016x}"))` — um basename de 16 hex fixos, sem separador. Todas as
chaves influenciáveis pelo cliente (`album:{album_id}`, `cover:{album_id}:{px}`,
`search:{q}`, `plname:{id}`, `pltracks:{id}:{off}`, `artist:{id}`) passam por essa
única função; `get/put/get_bytes/put_bytes/remove` a compartilham. Um `album_id`
hostil como `../../etc/passwd` vira um hex opaco e **não escapa** do state dir. Não há
caminho de disco montado a partir de chave crua. **`[verificado]`**

### Top-5 (vira backlog direto)

1. **GS-01 (P1)** `src/dcgi.rs:811` — `urldecode` faz **panic** com `%` seguido de char UTF-8 multibyte (`?q=%€`); aborta o request e (via `handlecgi`) suja a resposta com o stderr do panic. **`[verificado]`**
2. **GS-02 (P2)** `src/spotify.rs:603` — ids do cliente interpolados **sem encode** na URL da Web API; `..` colapsa via crate `url`, redirecionando o GET autenticado para endpoints arbitrários (GET-only, mesma conta). **`[verificado]`**
3. **GS-03 (P2)** `src/dcgi.rs:287` — injeção de linha em gophermap: `%0a` no `uri=` do menu humano forja uma entrada de menu (phishing/link arbitrário); a API máquina neutraliza, o menu humano não. **`[verificado]`**
4. **GS-04 (P2)** `src/cache.rs:38` — cache de disco **sem eviction**; entradas keyed por input crescem sem teto no `emptyDir` até DiskPressure despejar o pod.
5. **GS-05 (P2)** `src/cache.rs:40` — escritas de cache **não-atômicas** (`fs::write` sem temp+rename) → leitura torn servida ao cliente sob concorrência na mesma réplica.

## Achados detalhados

### GS-01 — P1 — `src/dcgi.rs:811` — panic no `urldecode` com `%`+UTF-8 multibyte  `[verificado]`
`&s[i+1..i+3]` fatia uma `&str` por offset de byte, com guarda só de comprimento
(`i+2 < len`), não de char boundary. Um `%` literal seguido de char de 3-4 bytes
(`?q=%€`, `?q=%你`, `?uri=%😀`) faz `i+3` cair no meio do char → panic
("byte index is not a char boundary", reproduzido). No dcgi por-request isso aborta o
processo; geomyidae `handlecgi` **splica o stderr do panic no socket**, corrompendo a
resposta e violando o contrato raw-`.cgi`. Alcançável de qualquer cliente nas rotas
humanas `/spot/*?...` **e** na máquina `/spot/api/1/search?q=`.
**Fix:** checar `b[i+1]`/`b[i+2]` como `is_ascii_hexdigit()` sobre o array de bytes
(ou `s.get(i+1..i+3)`), tratando falha como `%` literal. **Esforço: S**

### GS-02 — P2 — `src/spotify.rs:603` — injeção de request na Web API por id não validado  `[verificado]`
`album/track/artist/playlist` id chega como o **resto inteiro** do seletor
(`dcgi.rs:139-151`); `path()` não valida, não remove `/`, não normaliza `..`. É então
interpolado **sem encode** em `/v1/albums/{album_id}` (e `:641/733/757/772/787/801/811/828`).
O crate `url` (via ureq) resolve dot-segments, então `/spot/album/../../v1/me` monta
`…/v1/albums/../../v1/me` → normaliza p/ `/v1/me`, redirecionando o GET autenticado
(bearer do dono) para endpoints Spotify que o app não expõe (`/v1/me` perfil/email/país,
`/v1/me/player`, `/v1/me/following`). Limitado: `..` não cruza scheme/host, `?` é
pré-removido, só handlers de leitura (GET), conta do próprio dono, LAN-only. Ainda é
info-disclosure real por input não validado.
**Fix:** validar todo id contra base62 (`^[A-Za-z0-9]+$`) na borda dcgi/api, ou
percent-encodar cada id como um único segmento; rejeitar o resto. **Esforço: S**

### GS-03 — P2 — `src/dcgi.rs:287` — injeção de linha em gophermap no menu humano  `[verificado]`
O handler humano de play ecoa o `uri`/`context_uri` url-decodado numa info line
(`info(clip(&uri))` :287; contexto :264). O serializer do gopher-core escapa só `|` —
**não** remove `\n`/`\r`, e `latin1::encode` passa `\n` (0x0A) como identidade. Então
`?uri=x%0a[h|Click|URL:http://evil|server|port]` decoda o `%0a` para newline e os bytes
forjados viram uma **segunda linha física** `.gph` — uma entrada de menu type-h/type-1
injetada (phishing / seletor arbitrário). Reachable inclusive no arm `mock(...)`
(:296), que ecoa o `uri` sem chamar a API quando o Secret OAuth não está configurado.
A API máquina **não** é afetada: `api::kv` (`api.rs:455-465`) mapeia `\t\n\r`→espaço;
só o caminho humano não tem essa guarda.
**Fix:** neutralizar `\t\r\n` (e demais C0) em qualquer string derivada do cliente
antes de `info()`/`link()`, espelhando `api::kv`, ou no serializer do gopher-core.
**Esforço: S**

### GS-04 — P2 — `src/cache.rs:38` — cache de disco sem eviction, crescimento sem teto
`put/put_bytes` só criam arquivos; `get()` trata entrada expirada como miss mas **nunca
apaga**, e o único `remove` (`:71`) só busta a chave fixa `now_snapshot`. Cada query de
busca (`search:<q>`), álbum/artista/playlist navegado e capa JPEG (`cover:<id>:<px>`,
dezenas de KB) cria um arquivo permanente em `/var/cache/spot`. O volume é
`emptyDir: {}` (`deploy/gopher-server.yaml`) **sem `sizeLimit`**, puxando de ephemeral
storage do node sem limite. Em dias de uso real cresce até o kubelet bater DiskPressure
e despejar o pod (o limite de 128Mi de memória não ajuda em disco). Sem teste de eviction.
**Fix:** reaper best-effort (`read_dir` + remover expirados, oportunístico no `put`
ou sidecar cron) e/ou `emptyDir.sizeLimit`. **Esforço: M**

### GS-05 — P2 — `src/cache.rs:40` — escritas não-atômicas, leitura torn (confiança: média)
`put/put_bytes` usam `std::fs::write` (create-truncate + write) sem temp+rename e sem
lock. geomyidae gera **um processo dcgi por request**, então requests concorrentes na
mesma réplica compartilham `/var/cache/spot`. Na virada de TTL vários polls `/now`
dão miss simultâneo, cada um busca upstream (progress/ts ligeiramente distintos) e cada
`store_now` escreve o **mesmo** arquivo `now_snapshot` concorrentemente; uma leitura
concorrente pode observar arquivo truncado/interleaved — split falho (miss benigno) ou
doc `/now` truncado entregue ao cliente. Mesma corrida em `access_token` no cold start
→ token torn → 401 → doc de erro visível.
**Fix:** escrever em temp único no mesmo dir e `std::fs::rename` sobre a chave
(replace atômico); leitores sempre veem old ou new completo. **Esforço: S**

### GS-06 — P2 — `src/dcgi.rs:704` — overflow de u32 na paginação com `?offset=` adversarial (confiança: baixa)
`offset` é parseado direto do query em `u32` (`:200-203`, `unwrap_or(0)`, sem clamp
superior). Em `append_pager`, `has_next` computa `offset + (shown as u32)`. Com
`?offset=4294967290` e página não-vazia (`shown>0`), a soma faz overflow: panic em debug,
wrap silencioso no release que de fato roda, gerando link de pager bogus. Na prática o
Spotify retorna página vazia em offsets absurdos (`shown=0`, sem overflow), então só
dispara se algum impl retornar itens perto de MAX — daí a baixa confiança, mas o
`user→u32` sem clamp com add sem checagem é gap latente real.
**Fix:** `offset.saturating_add(shown as u32)` (e/ou clampar o offset parseado).
**Esforço: S**

### GS-07 — P3 — `src/spotify.rs:422` — sem single-flight no refresh de token (confiança: média)
`access_token()` checa cache em disco e, no miss, POSTa o endpoint de token
incondicionalmente. No start do pod / logo após expiração, um burst de polls `/now`
concorrentes (2 réplicas, vários processos por-request cada) dá miss independente e cada
um dispara um POST refresh. O endpoint de token é rate-limited; uma tempestade pode
tomar 429, quebrando auth brevemente até a primeira escrita vencer. Auto-corrige mas é
desperdício e pode amplificar um outage.
**Fix:** single-flight best-effort: `flock` num lockfile antes de refrescar, re-checar o
cache após adquirir. **Esforço: M**

### GS-08 — P3 — `docker/audio-stream-entrypoint.sh:197` — respawn da chain live a cada 2s, invisível ao liveness (confiança: baixa)
Os três loops (silence/drainer/live) têm backoff fixo (`sleep 2/1/2`), então **não há
flap de CPU**. Mas o backoff é constante (não-exponencial, sem teto): se a source
Icecast do ffmpeg é persistentemente rejeitada enquanto o Icecast segue de pé (senha de
source errada, slot esgotado, mount mal-config), o loop respawna a chain inteira —
incluindo re-auth do librespot no Spotify — a cada ~2s indefinidamente. O liveness probe
é **TCP :8000 apenas**, que fica verde porque o Icecast vive; esse estado
wedged-but-looping nunca reinicia e pode virar tempestade de login no librespot.
**Fix:** backoff exponencial com teto em falhas rápidas repetidas, e/ou contador de
falhas consecutivas que sai não-zero (container reinicia) quando a chain live não
estabelece source apesar do Icecast up. **Esforço: M**

## Confirmado SEGURO (auditado, sem achado)

- **Path traversal em disco** — refutado por hashing FNV (veredito acima). `[verificado]`
- **`urlencode()` (`spotify.rs:875`)** — allowlist base62 + `-_.~` percent-encoda CR/LF/`#`/`?`/`&`/espaço corretamente; `search q` e `queue/add uri` passam por ele.
- **`context_uri`/`play`** — chegam ao Spotify só dentro de body JSON (`serde_json`, JSON-escaped).
- **`api::kv`** — neutraliza `\t\r\n`→espaço, então a API máquina não é line-injectável.
- **Entradas numéricas** — `offset u32`, `volume 0-100`, `seek >=0`, `cover size ∈{64,300,640}` são parse-and-bound, sem panic (exceto o add do pager, GS-06).
- **Timeouts** — toda chamada outbound no módulo net passa por `self.agent` (deadline único `HTTP_TIMEOUT_SECS=10`); nenhum bypass no caminho de request. O único `ureq::post` sem timeout é `main.rs:208` (oauth-init, CLI, fora do request path).
- **Regra do stderr** — nenhum `eprintln!/dbg!` alcançável em `/spot/api/*`; os `eprintln!` existentes são do `main.rs` (usage) e do módulo oauth (CLI). O único vetor de corrupção stderr é o panic do GS-01.
- **Refresh de token entre réplicas** — cada réplica tem seu próprio `emptyDir`; não há compartilhamento de arquivo cross-réplica. As corridas (GS-05) são intra-réplica.

## Perf (lado servidor)

Nenhum achado exigiu timing de cluster: as questões do servidor são de
correção/robustez (GS-04 é disco, não latência), não de latência de request. O
micro-cache de `/now` (chave fixa `now_snapshot`, sobrescrita) é limitado e não cresce.
